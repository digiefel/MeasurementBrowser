# Figure Scripts

Figure scripts are generated by the MeasurementBrowser GUI to give you a reproducible Julia starting point for publication figures.

They do not draw the final figure for you. They resolve measurements, load them through MeasurementBrowser, analyze them, and leave you with a `data` object that you can use in normal Julia plotting code.

The generated script starts with `using MeasurementBrowser`, so run it in an environment where this package is available, for example:

```bash
julia --project /path/to/generated_script.jl
```

## GUI Workflow

1. Select measurements in the existing MeasurementBrowser panels.
2. Open `Figure Script` from the menu bar.
3. Choose an output directory.
4. Enter a script name.
5. Create one or more groups from the current selection.
6. Generate the script.

## What The Script Contains

The generated script defines four top-level values:

- `root_path::String`
  The measurement root that MeasurementBrowser will scan when the script runs.
- `project`
  The project parser/loader, for example `MeasurementBrowser.RUO2_PROJECT`.
- `groups::Vector{MeasurementBrowser.NamedMeasurementGroup}`
  The named measurement sets you selected in the GUI.
- `data::Dict{String,Vector{NamedTuple}}`
  The resolved and analyzed data, keyed by group name.

## What A Group Is

`NamedMeasurementGroup` is a plain Julia struct with two fields:

- `name::String`
- `measurement_ids::Vector{String}`

It has no special methods or hidden behavior. Use normal field access like `group.name` and `group.measurement_ids`.

The group name becomes the key in `data`:

```julia
data["control"]
data["fatigued"]
```

The order of `measurement_ids` is preserved in `data[group_name]`.

## What A Measurement ID Is

Each entry in `measurement_ids` is an exact MeasurementBrowser measurement id.

- A plain file id usually points to one measurement file.
- An id ending in `#cycle=...` points to one logical fatigue-cycle measurement expanded from a file.
- An id ending in `#split=...` points to one logical split measurement expanded from a file.

Do not casually rewrite these ids by hand. If you need a different set of measurements, regenerate the script or edit carefully.

## What `data` Contains

`data` is not a table or a dataframe.

Each `data[group_name]` entry is a `Vector` of plain named tuples.

Each entry has four fields:

- `measurement::MeasurementInfo`
- `plot_params::Dict{Symbol,Any}`
- `loaded`
- `analyzed`

`loaded` and `analyzed` are project-specific objects returned by MeasurementBrowser’s existing load/analyze pipeline.
These are also plain Julia values. Use normal field access like `entry.measurement` or `entry.analyzed`.

## Basic Usage

```julia
include("my_figure_script.jl")

control = data["control"]
first_control = first(control)

first_control.measurement
first_control.plot_params
first_control.loaded
first_control.analyzed

[entry.analyzed for entry in control]
[entry.measurement.clean_title for entry in control]
```

You normally build your final figure from the `analyzed` or `loaded` fields, depending on what the project returns for that measurement type.

## Minimal Example

```julia
using CairoMakie
include("my_figure_script.jl")

control = data["control"]

fig = Figure()
ax = Axis(fig[1, 1])

ys = [entry.plot_params[:temperature_K] for entry in control if haskey(entry.plot_params, :temperature_K)]

scatter!(ax, 1:length(ys), ys)

save("figure.pdf", fig)
```

Replace that example with the real fields for your measurement type. The important point is that the script already gives you resolved measurements and project-processed data in `data`.
