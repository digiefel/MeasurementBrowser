# Figure Scripts

Figure scripts are generated by the MeasurementBrowser GUI to give you a reproducible Julia starting point for publication figures.

They do not draw the final figure. They load the source-file union for the groups you defined in the GUI, expand those files into logical measurements, run MeasurementBrowser's normal load/analyze pipeline, and expose the result as `data`.

Run a generated script in an environment where MeasurementBrowser is available:

```bash
julia --project /path/to/generated_script.jl
```

## GUI Workflow

1. Select measurements in the existing MeasurementBrowser panels.
2. Open `Figure Script` from the menu bar.
3. Choose an output directory.
4. Enter a script name.
5. Create one or more groups from the current selection.
6. Generate the script.

## What The Script Contains

The generated script defines five top-level values:

- `root_path::String`
  The measurement root that MeasurementBrowser uses for metadata and validation.
- `project`
  The project parser/loader, for example `MeasurementBrowser.RUO2_PROJECT`.
- `source_files::Vector{String}`
  The union of source files needed by all stored groups.
- `groups::Vector{MeasurementBrowser.NamedMeasurementGroup}`
  The fixed named groups defined in the GUI.
- `data::MeasurementBrowser.FigureScriptData`
  The loaded/analyzed figure-script data object.

## What A Group Is

`NamedMeasurementGroup` has two fields:

- `name::String`
- `filter::MeasurementBrowser.MeasurementGroupFilter`

The group name is the lookup key in `data`:

```julia
data["control"]
data["fatigued"]
```

Only groups that were stored in the generated script exist. `data["new_name"]` does not create a new filter on the fly.

## How Group Filters Work

`MeasurementGroupFilter` is an OR of `MeasurementFilterClause` values.

Each clause can match on:

- exact source file
- exact measurement kind
- exact device path or device-path prefix
- exact parameter equalities

Groups are inferred from the GUI selection when you generate the script. They are stored as fixed filters, not as serialized measurement ids.

## What `data` Is

`data` is not a table and not a dataframe.

`FigureScriptData` has these useful fields:

- `data.source_files::Vector{String}`
- `data.measurements::Vector{MeasurementBrowser.FigureMeasurement}`

Each `FigureMeasurement` has:

- `measurement::MeasurementInfo`
- `parameters::Dict{Symbol,Any}`
- `loaded`
- `analyzed`

`data["group_name"]` returns `Vector{MeasurementBrowser.FigureMeasurement}` for that stored group.

## Basic Usage

```julia
include("my_figure_script.jl")

control = data["control"]
first_control = first(control)

first_control.measurement
first_control.parameters
first_control.loaded
first_control.analyzed

[entry.analyzed for entry in control]
[entry.measurement.clean_title for entry in control]
```

`parameters` is the merged parameter dictionary for that logical measurement:

- `measurement.device_info.parameters`
- merged with `measurement.parameters`

## Minimal Example

```julia
using CairoMakie
include("my_figure_script.jl")

control = data["control"]

fig = Figure()
ax = Axis(fig[1, 1])

ys = [entry.parameters[:temperature_K] for entry in control if haskey(entry.parameters, :temperature_K)]

scatter!(ax, 1:length(ys), ys)

save("figure.pdf", fig)
```

Replace that example with the real fields for your measurement type. The important point is that the generated script already gives you fixed named groups over project-processed logical measurements.
